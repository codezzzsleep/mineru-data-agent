# MinerU Data Agent 技术报告

## 摘要

MinerU Data Agent 是面向 MDIC 2026 赛道二的数据智能体系统。项目以 `data-agent` 命令行工具作为稳定交付入口，围绕 MinerU 的 PDF/图片解析能力，补充任务规划、结构化抽取、质量校验、自动恢复、可追溯日志、检索导出和可选 LLM tool-calling Agent。

系统目标不是替代 MinerU，而是在 MinerU 输出之上形成一个可复跑、可审计、可扩展的 Data Agent：输入复杂文档和自然语言任务，自动选择处理路径，调用解析、抽取、校验、恢复和导出模块，最终生成结构化结果、运行日志和证据材料。

本文是提交给评委的完整技术报告，覆盖：

- Data Agent 整体设计方案
- 任务执行机制
- 数据处理与工具调用能力
- 系统性能与稳定性
- 典型任务执行示例
- 系统适用场景与应用价值
- 当前边界与后续改进方向

## 1. 项目定位与问题定义

真实业务中的财报、合同、行业标准、扫描件、网页报告和流程文档通常具有以下困难：

- PDF 版面复杂，表格、标题、正文和图像交错。
- 长文档存在跨页上下文和分片处理问题。
- 财报类文档包含密集数字、合计行、小计行和异常值。
- 扫描件和 OCR 文本会出现乱码、错字、断行和缺页级来源。
- 合同、标准和流程文档需要保留条款、章节、风险提示和来源证据。
- 单纯 LLM 直接抽取容易出现幻觉、漏证据和数字解释错误。

本项目采用“解析工具 + Agent 编排 + 质量审计”的路线：用 MinerU 负责 PDF/图片解析，用轻量原生模块处理 HTML/DOCX/PPTX，用 Agent 层完成任务画像、结构化抽取、质量检查、恢复重试、日志记录和 retrieval 导出。

## 2. 整体设计方案

系统采用分层架构：

| 层级 | 作用 | 主要输出 |
| --- | --- | --- |
| Task Planner | 根据任务、文件名和 profile 配置推断处理场景，生成执行控制字段 | `execution_control`、`agent_action_plan` |
| MinerU Adapter | 调用 MinerU 在线 Agent API 或本地 MinerU CLI 解析 PDF/图片 | Markdown、content list、MinerU artifact |
| Native Extractors | 处理 HTML、DOCX、PPTX 等非 PDF 输入 | Markdown、内容块、表格、章节 |
| Structured Extractor | 从解析结果中抽取章节、表格、键值、数字事实、语义信号 | `extracted` |
| Quality Validator | 检查空结果、乱码、页级来源、合计行、弱结构等风险 | `quality` |
| Recovery Orchestrator | 根据质量问题执行 text cleanup、OCR retry、CLI fallback，并择优 | `recovery_decision` |
| Retrieval Exporter | 导出适合检索和 RAG 的 chunks | `retrieval/` |
| Trace Logger | 记录每一步、工具调用、耗时、错误摘要 | `trace.json` |
| CLI Layer | 提供 `run`、`batch`、`agent-run` 三个入口 | 运行目录和报告 |

核心输出目录包含：

- `result.json`：机器可读结构化结果。
- `trace.json`：步骤、工具、耗时和错误日志。
- `summary.md`：人工可读摘要。
- `retrieval/retrieval_chunks.jsonl`：检索 chunks。
- `retrieval/retrieval_manifest.json`：检索导出元数据。

## 3. 任务执行机制

### 3.1 单文档执行流程

`data-agent run` 的执行流程如下：

1. 接收输入文件、自然语言任务、profile、runner、backend 和 method。
2. 根据任务和文件名推断 profile，例如财报、合同/标准、流程图、低质量 OCR 或通用文档。
3. 生成 `execution_control.planning_rationale`，记录 profile、runner、backend、method 和恢复策略的选择原因。
4. 若启用 LLM，先执行解析前调度，让模型提出 profile、backend、method、语言、目标 schema 和复核重点建议；系统只应用白名单内且未被用户锁定的建议。
5. 生成 `agent_action_plan`，记录子任务、候选工具、动态选择和 replan triggers。
6. 根据文件类型调用 MinerU 或原生解析器。
7. 构造结构化视图：章节、表格、键值对、数字事实、语义信号、跨页引用、字段证据。
8. 执行质量校验，生成质量分和 issue codes。
9. 若命中可恢复风险，执行 text cleanup、OCR retry 或 CLI fallback，并按质量择优。
10. 生成 `recovery_decision`，记录每次 attempt、失败原因、最终选中结果和剩余风险。
11. 导出 retrieval chunks。
12. 写出 `result.json`、`trace.json`、`summary.md` 和 retrieval 文件。

### 3.2 批处理机制

`data-agent batch` 接收 JSON manifest，逐个运行任务。单个任务失败不会中断整批，最终写出 `batch_report.json`，包含每个任务的状态、run id、输出路径、质量分和错误摘要。

### 3.3 Live LLM Agent 机制

`data-agent agent-run` 是真实 provider tool-calling Agent 路径，需要配置 ModelScope 或 DeepSeek 等 OpenAI-compatible provider key。

新版 live Agent 采用 skill-guided 机制：

1. 模型必须先调用 `select_skill`，选择高层策略。
2. 可选 skill 包括财报合计核验、not_found 防幻觉、文本恢复后抽取、合同条款复核、流程风险复核和通用结构化抽取。
3. 文档解析后，模型必须调用 `validate_answer`。
4. `validate_answer` 检查证据支持、未支持数字、简单算术矛盾和 selected-skill 下的 not_found 冲突。
5. `finalize` 只能使用已经通过验证的同一答案和同一证据列表。

这一路径保留 LLM 自主选择工具和策略的空间，同时用 skill、工具顺序和答案验证约束随意发挥。

当前保存的 `submission_artifacts/agent_live_cases/` 是旧版 provider 包，包含 8 次 ModelScope Qwen3 尝试、4 次 tool-call completion、2 次人工复核 answer-quality pass。该包生成于更严格的 skill/validation gate 之前，因此只作为旧版真实 provider trace，不声称新版 gate 已完成 provider rerun。

## 4. 数据处理与工具调用能力

### 4.1 支持的输入类型

| 类型 | 处理路径 | 说明 |
| --- | --- | --- |
| PDF/图片 | MinerU 在线 Agent API 或本地 MinerU CLI | 用于版面解析、OCR、Markdown 和 content list 输出 |
| HTML | 原生 HTML extractor | 保留标题、段落、列表、表格和链接文本 |
| DOCX | 原生 Office extractor | 读取正文、表格和基本结构 |
| PPTX | 原生 Office extractor | 读取 slide 文本、表格和基本页面结构 |

### 4.2 MinerU 调用方式

CPU 或快速验证环境可使用在线 Agent API：

```bash
data-agent run --runner agent-api --input demo.pdf --out runs/pdf_api --task "解析 PDF 并输出结构化结果和质量日志"
```

需要页级 provenance 和完整中间 artifact 时使用本地 MinerU CLI：

```bash
data-agent run --runner cli --input demo.pdf --out runs/pdf_cli --task "解析财报 PDF，抽取表格、关键数字并检查合计行" --backend pipeline --method auto
```

本地 CLI 路径会保留 MinerU Markdown、content list、middle/model JSON、layout PDF、span/origin PDF 和图片 artifact。

### 4.3 结构化抽取

系统从解析结果中生成：

- `sections`：章节和标题层级。
- `tables`：Markdown 表格和 HTML table 解析结果。
- `key_values` / `key_value_map`：键值候选。
- `numeric_facts`：数字事实和所在文本。
- `semantic_signals`：日期、异常、建议、风险等语义信号。
- `cross_page_references`：简单跨页引用线索。
- `field_evidence`：字段证据、来源文本、confidence proxy、页/行/块来源。

轻量抽取层不只做冒号分割，还补充了两列表格键值、多行键值、标题后紧邻段落、简单跨页引用和 HTML table 解析。

### 4.4 质量校验

当前质量检查包括：

- 空结果检查。
- 编码噪声和乱码检查。
- 页级 provenance 检查。
- profile 预期结构检查。
- 财报合计行/小计行核验。
- 合同和标准文档章节结构检查。
- 任务要求日期、异常、建议时的语义信号检查。
- 长文本缺少标题、键值或表格时的弱结构提示。
- 审计型任务的 strict page provenance gate。

质量状态分为 `pass`、`pass_with_warnings` 和 `needs_review`。这避免把存在 warning 的输出包装成完全正确。

### 4.5 自动恢复

恢复策略包括：

- `text_cleanup`：清理乱码、异常空白和常见 OCR 噪声。
- `ocr_retry`：对 PDF/图片尝试 OCR 路径重跑。
- `cli_fallback`：在线 API 缺少页级 provenance 时切换到本地 CLI artifact。

恢复后系统会比较初始结果和恢复结果，按质量状态、质量分、issue penalty 和 warning 数择优。所有尝试都会记录在 `recovery_decision.attempts` 和 `trace.json` 中。

## 5. 系统性能与稳定性

### 5.1 可复现性设计

项目提供：

- `pyproject.toml` 依赖声明。
- CLI-first 入口：`data-agent run`、`data-agent batch`、`data-agent agent-run`。
- Dockerfile 和 docker-compose。
- GitHub Actions 测试 workflow。
- 每次运行落盘 `result.json`、`trace.json`、`summary.md`。
- 失败运行也保留失败态 trace。
- API key 和本机路径脱敏。
- 提交 zip inventory 测试，检查必要文件和本机路径泄露。

| p001_020 | 1-20 | completed | 23 |
| p021_040 | 21-40 | completed | 26 |

## 7. 系统适用场景与应用价值

### 7.1 财报和审计材料

适合抽取财务表格、关键数字、合计行、小计行和异常提示。质量校验能把数字风险转化为可审计 issue code，减少纯 LLM 抽取时的数字幻觉。

### 7.2 合同和行业标准

适合抽取合同编号、签署日期、章节条款、责任主体、异常处理和合规矩阵。字段证据和 provenance 便于人工复查。

### 7.3 扫描件和低质量 OCR

适合识别乱码、断行、低质量文本和 OCR 风险。系统可先清理文本，必要时提示 OCR retry 或人工复核。

### 7.4 流程和工程资料

适合抽取流程步骤、异常节点、风险动作和负责人信息。对于包含复杂图像的工程资料，当前系统会保留图像 artifact 和视觉复核提示。

### 7.5 知识库与 RAG 入库

系统会导出 retrieval chunks、manifest 和 quality report，适合后续进入向量库、全文检索或企业知识库。

## 8. 成本、速度与精度平衡

项目提供多种运行路径：

| 路径 | 成本特点 | 速度特点 | 精度/证据特点 |
| --- | --- | --- | --- |
| Native HTML/DOCX/PPTX | 无外部解析成本 | 快 | 适合结构化文本和回归测试 |
| MinerU 在线 Agent API | 无本地 GPU 要求 | 适合 CPU 环境快速验证 | 可能缺页级 provenance |
| 本地 MinerU CLI | 需要本地模型/GPU/CPU 资源 | 取决于硬件 | artifact 最完整，适合审计 |
| LLM preplan/review | 有 token 成本 | 取决于 provider | 增强任务理解和风险复核 |
| Live tool-calling Agent | 成本最高且依赖 provider 稳定性 | 较慢 | 展示自主工具选择和证据校验 |

保存的成本模型位于 `submission_artifacts/cost_model/`。其中价格是可替换的 2026 年 5 月场景假设，生产规划前应替换为当前 provider 报价。

## 10. 安全与脱敏

- LLM API key 只从环境变量读取。
- `result.json`、`trace.json`、summary 不写入 API key。
- DeepSeek/ModelScope 错误摘要会过滤 Bearer token 和 `api_key=` 参数。
- MinerU 在线 Agent API 下载 URL 会过滤 `token=`、`access_token=`、`signature=`、`X-Amz-*` 等签名字段。
- 提交材料收集脚本会把本机路径替换成 `<PROJECT_ROOT>`、`<USER_HOME>` 或 `<MINERU_ROOT>`。

## 11. 当前边界

项目当前已经形成完整 CLI 工程包，但仍保留以下边界：

- 保存的 live tool-calling Agent provider 包是旧版 pre-skill-gate 证据，不声称新版 gate 已完成 provider rerun。
- 带标注评测包含自建 fixture 和轻量公开 PDF 标签，不等同第三方 OCR benchmark。
- 长文档证据展示在线 API 分片编排，不等同 GPU 长文档 benchmark。
- HTTP API 是本地可选 wrapper，不是主要交付面。
- 对复杂工程图、手写签章重叠、表格逐格准确率和 OCR 字符级准确率的公开 benchmark 仍需后续扩展。

## 12. 结论

MinerU Data Agent 的核心价值是把 MinerU 文档解析能力组织成一个可复跑、可审计、可恢复的 Data Agent 工程系统。它通过 CLI-first 方式提供稳定评审入口，通过 `result.json`、`trace.json`、`summary.md` 和 retrieval artifacts 保证输出可追溯，通过质量检查和恢复机制降低复杂文档处理中的空结果、乱码、缺页级来源和数字错误风险。

在当前提交中，系统更强的是工程可复现性、质量审计和多格式处理；Live LLM Agent 自主性已有 skill-guided 工具闭环设计，但保存的 provider 证据仍偏少，后续应重点补齐新版 skill-gated live rerun、公开 benchmark 和更复杂真实长文档案例。
